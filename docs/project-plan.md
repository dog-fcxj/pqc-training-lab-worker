# PQC Training Lab 项目规划

## 1. 项目定位

当前项目不是一个缺少页面样式的前端半成品，而是一个已经具备视觉语言、但内容层和交互层尚未落地的 PQC 展示站。

建议将其正式定位为：

`PQC 教学与迁移决策实验室`

面向对象：

- 网络工程师
- 安全工程师
- 架构师
- 需要快速理解 PQC 工程影响的研发团队

项目目标：

- 用一套统一的视觉风格讲清 PQC 为什么是现在的问题
- 让用户能够比较 ML-KEM、ML-DSA、SLH-DSA、HQC 的角色与代价
- 让用户能从 TLS、IKEv2、PKI、origin 等工程场景理解迁移影响
- 在不引入复杂后端的前提下，做成可部署、可演示、可扩展的静态站点

## 2. 当前项目现状

当前仓库的特点：

- 技术栈极轻：Cloudflare Worker + 静态资源
- 页面结构明确：Hero / Principles / Algorithms / Deployment 四屏
- 视觉基调已经成型：深色科技风、全屏翻页、卡片式内容
- 数据驱动能力不足：`public/data.js` 仅支持简短文案展示
- 详情层未成体系：只有算法详情有基础内容，其余类型均为占位
- 交互深度不足：缺少真正可解释概念的图示、流程、比对和迁移建议

当前主要问题不是 UI，而是：

1. 内容模型过薄
2. 信息架构与页面结构没有完全对齐
3. 交互组件大多仍是占位
4. 页面缺少教学闭环和工程决策闭环
5. 移动端和可访问性没有纳入设计

## 2.1 演进背景确认

根据当前仓库提交记录，可以确认这个项目经历了三段明显演进：

1. 初版直接承接了 `pqc-sharing-kit` 的知识型 demo 结构
2. 中期引入了更强调视觉表现的版本，并持续融合新的前端风格
3. 后期提交几乎集中在“前端样式重构、视觉稳定、模块化拆分、全屏沉浸式表达”上，内容被有意弱化

这意味着当前项目的真实状态应理解为：

- 它不是一个普通意义上“前端没写完”的站点
- 它是一个经过多轮视觉重写后，刻意把内容先放到次要位置的样式优先版本
- 因此后续规划不应推翻现有视觉方向，而应以“把知识和工程判断重新灌回这套 UI 骨架”为主

从提交历史看，后期关键阶段包括：

- `71f36a2`
  完整重写为沉浸式实验室 UI
- `389ffad` 到 `a7786e4`
  多轮背景、导航、视觉稳定性和布局修正
- `ea07ae4`
  将页面重构为模块化结构，基本形成当前代码形态

因此，本规划中的核心策略保持不变，但语义需要更准确：

- 不是“补全一个半成品前端”
- 而是“在已经满意的视觉重构成果上，重新注入内容与交互闭环”

## 4. 推荐产品结构

建议保留现有四屏，但重新定义为更明确的用户路径：

### 4.1 Hero

目标：

- 用一句话说明站点价值
- 在首屏说明 PQC 为什么要现在关注
- 给出三到四个最重要的定量信号

建议内容：

- PQC 定义
- HNDL 风险
- NIST 关键时间点
- 公开部署进展摘要

建议交互：

- 首屏 CTA 分别跳转到算法页和工程场景页
- 增加一组小型事实卡片，而不是只有大标题与概念文案

### 4.2 Principles

目标：

- 不是做数学证明，而是做三种路线的认知建立
- 让用户理解格基、哈希基、码基为什么不同

建议内容：

- 每种原理一组简化解释
- 对应的工程含义
- 对应的代表算法

建议交互：

- 选择原理后切换动画演示
- 同屏展示“直观解释 / 工程意义 / 代表算法”
- 详情层展示更深入的结构说明

### 4.3 Algorithms

目标：

- 建立算法比较框架，而不是只展示名字

建议内容字段：

- family
- role
- sizes
- facts
- pathSteps
- formula
- takeaway

建议交互：

- 左侧算法列表，右侧详情面板
- 增加“对象尺寸 / 接口 / 工程角色”的固定比较区
- 点击算法卡片时不应只打开同一弹层，而应切换详情状态

### 4.4 Deployment

目标：

- 将“协议开销”与“迁移判断”串起来

建议拆成两个子区：

- Handshake Lab
- Migration Lab

建议内容：

- TLS classical vs hybrid 对比
- IKEv2 + ML-KEM 的对象与部署压力
- 系统画像清单
- 自动生成迁移关注项

建议交互：

- 通过系统画像选项生成迁移建议卡片
- 不只展示摘要，还要展示“为什么是这个建议”

## 5. 建议实施路线

为避免重新注入内容时破坏现有满意的视觉结果，建议采用“小步快跑、阶段冻结、随时可回退”的方式推进。

总原则：

- 每个阶段只解决一类问题，不跨阶段混改
- 每完成一个阶段就做一次可运行检查
- 每个阶段保留清晰的回退锚点
- 在视觉、导航、详情层三个高风险区域优先做低侵入改造

## 阶段一：内容模型重建

目标：

- 从展示型数据升级为页面驱动数据

输出：

- 新版 `public/data.js` 或拆分后的 `public/data/*.js`
- 支持 Hero、Principles、Algorithms、Handshake、Migration 五类内容

重点：

- 将 `pqc-sharing-kit/demo/public/app.js` 中的数据迁入当前项目
- 统一命名与字段结构
- 为详情层预留 `type + id` 查询模式

执行方式：

1. 先保留当前 `public/data.js` 的对外导出名，避免首轮修改牵动所有组件
2. 在原字段兼容的基础上补充新字段，而不是一步到位删除旧字段
3. 先让旧页面在新数据上继续可运行，再逐步改组件消费方式

阶段验收：

- 页面可正常加载
- 当前四屏不因数据结构扩展而报错
- 详情层已具备读取更丰富数据的基础

回退措施：

- 若新数据结构导致多个组件联动报错，先回退为“旧字段保留 + 新字段附加”的兼容模式
- 若新增数据量过大导致维护困难，拆分为 `hero / principles / algorithms / handshake / migration` 五个模块文件
- 若 detail schema 设计不稳定，先只统一数据格式，不立即统一详情层渲染逻辑

## 阶段二：页面模块补完

目标：

- 让每一屏具备清晰教学目标和交互价值

输出：

- Hero 指标卡
- Principles 动态演示
- Algorithms 对比详情
- Handshake 定量视图
- Migration 结果面板

重点：

- 替换现有占位内容
- 将主要内容渲染与数据结构解耦

执行顺序：

1. Hero
2. Principles
3. Algorithms
4. Handshake
5. Migration

原因：

- 先完成认知入口，再完成核心比较，再进入工程判断

阶段验收：

- 每一屏都能回答一个明确问题
- 页面内不再出现纯占位提示
- 用户不打开详情层也能完成基础阅读

回退措施：

- 若某一屏改造幅度过大，优先保留现有视觉骨架，只替换内容区
- 若可视化组件不稳定，先降级为静态图例 + 结构化说明
- 若四屏同时推进导致样式互相污染，按屏幕独立收敛，暂缓共享 class 抽取

## 阶段三：详情系统重构

目标：

- 把临时拼 HTML 的详情层升级为可扩展的统一内容容器

输出：

- 统一 detail registry
- 原理、算法、握手场景、迁移结果都可复用同一详情机制

重点：

- 详情结构可配置
- 保持原有视觉语言

执行方式：

1. 先抽象 detail 数据入口
2. 再抽象 detail 内容块类型
3. 最后统一 open / close / title / body 渲染流程

建议支持的内容块：

- summary
- factGrid
- bulletList
- stepList
- formulaBlock
- advisory

阶段验收：

- `algorithm`、`principle`、`handshake` 至少三类详情已统一
- `openDetail()` 不再需要长分支拼接 HTML

回退措施：

- 若统一 detail renderer 导致复杂度激增，先保留多类型 detail data，渲染层只统一外框
- 若弹层重构影响主页面交互，先冻结主页面功能，只修复弹层最小闭环
- 若出现视觉回归，优先回退弹层内部内容渲染，不回退弹层开合机制

## 阶段四：体验与质量补强

目标：

- 让站点从“能演示”提升到“可上线展示”

输出：

- 移动端适配
- 键盘与触控导航
- favicon / meta / footer 来源说明
- 基础性能与可访问性优化

执行方式：

1. 先处理导航可达性
2. 再处理移动端布局
3. 最后处理站点收尾项

阶段验收：

- 不依赖滚轮也能浏览
- 主要页面在手机宽度下可读
- favicon 与 meta 信息完整

回退措施：

- 若全屏翻页在移动端成本过高，可对移动端降级为普通纵向滚动
- 若 hash 同步与翻页状态冲突，先保留本地状态控制，后补 URL 同步
- 若移动端样式修复影响桌面端视觉，优先通过媒体查询隔离，不直接修改桌面基线样式

## 5.1 风险分级与改造策略

### 低风险任务

- 数据字段扩展
- Hero 文案增强
- favicon / meta / 页脚补充

策略：

- 可直接改主干

### 中风险任务

- Principles 与 Algorithms 的信息重排
- Migration 结果面板新增
- 详情层数据模型改造

策略：

- 一次只改一个页面
- 每次改动后做本地运行检查

### 高风险任务

- 全局导航逻辑
- 详情层统一渲染
- 移动端翻页机制
- 大规模样式抽离

策略：

- 必须分步骤提交
- 每一步都保留可回退中间状态
- 不在同一轮同时改逻辑和视觉结构

## 6. 技术方案建议

短期建议：

- 继续使用当前静态架构
- 保持 Cloudflare Worker 作为静态站部署外壳
- 先完成信息架构与交互补全

原因：

- 当前规模不需要引入前端框架
- 现有代码量很小，手工重构成本低
- 目前主要矛盾是内容与交互，不是状态管理

中期可选路线：

- 如果后续要引入更多组件、过滤器、状态同步、图表或中英文切换，再迁 React / Vite

不建议现在立刻做的事情：

- 不建议先重做 UI 风格
- 不建议先切换技术栈
- 不建议先上后端

## 6.1 知识扩展性设计

这个项目后续一定会继续加知识内容，因此扩展性不能只理解为“代码好维护”，还要理解为“新知识能低成本接入，不破坏现有页面”。

需要提前满足的扩展目标：

- 新增一个算法时，不需要重写算法页结构
- 新增一个协议场景时，不需要重写工程页布局
- 新增一类知识块时，不需要大改详情层
- 新增更深层资料时，主页面与详情层能分层承载
- 页面文案更新时，不需要改大量组件模板

建议采用三层结构：

### 第一层：知识数据层

职责：

- 只存事实、解释、步骤、公式、建议、来源标识

要求：

- 数据不夹杂布局细节
- 字段命名稳定
- 每类知识对象都有唯一 `id`
- 每类知识对象都能被详情层直接消费

建议对象类型：

- `heroMetric`
- `principle`
- `algorithm`
- `handshakeScenario`
- `migrationRule`
- `detailDocument`

### 第二层：页面映射层

职责：

- 决定某类数据出现在什么页面、以什么顺序出现

要求：

- 页面顺序和内容顺序分离
- 支持“主页面摘要 + 详情层深度内容”的两级映射

建议做法：

- 每个页面有自己的 section config
- section config 只引用数据 id，不直接写死文案

### 第三层：展示组件层

职责：

- 把结构化数据渲染为卡片、列表、步骤、公式块、建议卡

要求：

- 组件按内容块类型复用
- 尽量避免针对单个算法或单个场景写专用模板

## 6.2 建议的数据扩展规则

后续新增知识内容时，建议遵循以下规则：

1. 新知识优先加到数据层，而不是直接写进组件 HTML
2. 新增字段时尽量保持向后兼容
3. 新增内容优先复用已有内容块类型
4. 只有当现有内容块无法表达时，才新增新的 renderer 类型
5. 每条关键知识建议保留 `sourceKey` 或 `sourceNote`

推荐字段示例：

- `id`
- `title`
- `summary`
- `facts`
- `steps`
- `formula`
- `takeaway`
- `relatedIds`
- `sourceKey`
- `updatedAt`

## 6.3 可扩展内容块协议

建议详情层和页面正文尽量围绕固定内容块协议构建。

推荐块类型：

- `summary`
- `statList`
- `factGrid`
- `compareTable`
- `stepList`
- `formulaBlock`
- `quoteNote`
- `advisoryList`
- `linkList`

好处：

- 后续新增“FN-DSA”“证书链迁移”“CT/SCT 影响”“代码签名”等内容时，只需要提供新数据，而不一定要写新模板

## 6.4 后续可能扩展的知识方向

为了避免当前结构过于贴死现有四类内容，建议在模型上预留这些扩展方向：

- `signatureMigration`
  证书链、CT、SCT、代码签名、固件签名
- `protocolCase`
  TLS、IKEv2、QUIC、Zero Trust、Tunnel、内部 RPC
- `deploymentConstraint`
  HSM、网关、老 SDK、移动端、边缘节点、硬件加速
- `timelineEvent`
  NIST、IETF、Cloudflare 公开进展时间线
- `glossaryTerm`
  KEM、HNDL、NTT、rejection sampling、PMTU 等术语

当前版本未必全部实现，但数据模型不要把这些方向堵死。

## 7. 内容基线建议

站点事实口径建议统一以 `2026-04-07` 为基线日期，并在页脚注明资料来源类别：

- NIST 标准化与迁移项目
- IETF 草案状态
- Cloudflare 公开部署数据

同时建立一个维护原则：

- 页面中的关键数字必须能追溯到参考资料
- 教学比喻可以简化，但工程结论不能模糊
- 新增知识条目时要记录更新时间，避免旧口径和新口径混在一起

## 8. 风险与注意事项

主要风险：

- 内容一旦过度追求“炫技”，会削弱教学价值
- 继续使用全屏翻页时，移动端和可访问性容易出问题
- 如果仍使用内联样式拼装复杂页面，后续维护会迅速变差
- 详情层若继续用分支判断硬编码，扩展到第五类内容会变得混乱

对应建议：

- 先把内容层做厚，再打磨动效
- 页面组件样式逐步抽离出 class
- 所有展示模块优先走数据驱动

## 10. 结论

最合理的路线不是推翻重做，而是：

1. 复用参考项目中的内容主线与事实口径
2. 将当前项目升级为真正的数据驱动演示站
